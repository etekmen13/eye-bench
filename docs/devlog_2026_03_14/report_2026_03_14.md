# eye-bench: Current Project Summary

## Overview

`eye-bench` is a reproducible benchmark and model-comparison framework for the Noita eye glyph puzzle.

The working goal is **not** to guess plaintext directly. The goal is to determine which kinds of generative/cipher families can reproduce the structural fingerprints of the real Noita corpus under the accepted **trigram-token representation**.

In other words, the repo is being built to answer:

> What model family best matches the statistical and structural behavior of the Noita eye messages?

This is a much better contribution than ad hoc decipherment attempts because it turns community observations into executable tests.

---

## Current framing

The benchmark operates on the **canonical trigram-token corpus**, not on raw eye glyphs.

That means all statistics are computed over sequences of trigram token IDs, where a “unigram” means:

* one trigram token

and an “n-gram” means:

* an n-gram over trigram tokens

This gives a single clean representation shared by:

* the real Noita corpus
* synthetic corpora sampled from candidate model families

---

## What has been implemented

### Corpus layer

The repo now has a basic corpus pipeline:

* `Corpus` / `Message` schema
* corpus loader
* corpus validation
* conversion to integer arrays

This provides a canonical structured input for all downstream analysis.

---

### Invariants / benchmark feature extraction

A benchmark feature extraction layer has been implemented over trigram-token sequences.

#### Core information-theoretic features

* unigram entropy
* conditional entropy by order `k`
* lagged mutual information
* repeat rate by lag
* self-follow rate

#### Count-based structural features

* exact n-gram counts
* pairwise shared exact n-gram counts across messages

#### Isomorph / pattern-shape features

* first-occurrence pattern signatures
* isomorph window counts
* pairwise shared isomorph-window counts

#### Benchmark assembly

A `compute_corpus_stats(...)` function computes pooled and pairwise benchmark statistics for a corpus and produces a structured `CorpusStats` object.

The true Noita corpus stats are now also exported to:

* `data/processed/noita_corpus_stats.json`

---

### Scoring

A comparison/scoring layer has been built.

It currently compares two `CorpusStats` objects using a weighted sum of component distances, including:

* entropy terms
* lag dependence terms
* n-gram distribution differences
* shared-structure differences

This is currently good enough for family ranking, though some parts should be improved later.

---

### Generator benchmarking

A generator evaluation pipeline has been implemented.

It supports:

* defining a list of generator families
* sampling many synthetic corpora over many seeds
* computing benchmark stats for each run
* scoring each run against the true Noita corpus
* ranking families by mean weighted distance

This is now the main experimental loop.

## Baseline families that have been tested

### 1. i.i.d. unigram baseline

A trivial synthetic generator was implemented that samples each token independently from a fitted unigram distribution.

Purpose:

* establish the weakest possible baseline
* test whether the Noita corpus contains structure beyond token frequencies

Result:

* clearly worse than Markov-1
* confirms that the Noita corpus is not well explained by a memoryless source

---

### 2. Markov-1 baseline

A first-order Markov generator was implemented.

Purpose:

* test whether shallow observed-token sequential dependence explains the corpus

Result:

* currently the best-performing baseline family
* substantially better than i.i.d.
* demonstrates that short-range sequential structure is real and important

This is now the strongest shallow baseline.

---

### 3. Markov-2 baseline

A sparse second-order Markov model was implemented.

Purpose:

* test whether adding more local context improves fit

Result:

* worse than Markov-1

A diagnostic analysis showed that this is explained by extreme sparsity:

* only a small fraction of possible 2-token contexts are observed
* most observed contexts are singletons or very low support
* during sampling, unseen-context fallback occurs on most steps

So Markov-2 was not just “a little noisy”; it was dominated by sparse estimation.

---

### 4. Markov-2 backoff / interpolated model

An interpolated/backoff Markov-2 variant was implemented to mitigate sparsity.

Purpose:

* test whether smoother higher-order local memory helps

Result:

* slightly better than raw Markov-2
* still clearly worse than Markov-1

Conclusion:

* pushing shallow observed-token Markov order upward is not the right direction on this corpus

---

### 5. Permutation-state v1

A first stateful cipher-like generator was implemented.

Model:

* latent source = fitted Markov-1
* hidden state = cyclic
* one permutation per state
* observed token emitted via state-dependent permutation

Purpose:

* test whether evolving remapping of a latent source improves fit

Result:

* competitive with Markov-1
* did not beat it over 50 seeds
* but showed that stateful remapping is not obviously a dead end

This was the first real move beyond pure observed-token Markov modeling.

---

### 6. Permutation-state v2

A stronger stateful family was implemented.

Model:

* latent source = fitted Markov-1
* state permutations = same permutation bank
* hidden state update = token-driven rather than purely cyclic
* default token-to-state-shift map = bucketed, to reduce variance

Purpose:

* test whether latent-token-driven state evolution improves over both Markov-1 and permutation-state v1

Result:

* better than permutation-state v1
* still worse than Markov-1 over 50 seeds

This suggests that simple evolving-remapping alone is not yet enough to beat the best shallow baseline.

## Current ranking over 50 seeds

1. `markov1` — **3.645607**
2. `perm_state_v2` — **3.768014**
3. `perm_state_v1` — **3.803394**
4. `markov2_backoff` — **4.091793**
5. `markov2` — **4.119441**
6. `iid_unigram` — **4.299769**

Interpretation:

* the Noita corpus definitely has more structure than i.i.d.
* shallow local dependence matters
* sparse higher-order local dependence does not help
* simple stateful permutation families are promising but not yet superior to Markov-1

## What the current results mean

## 1. The benchmark is working

The family ordering makes sense:

* i.i.d. is worst
* Markov-1 is much better
* sparse Markov-2 is penalized
* simple stateful models land in a plausible middle ground

That is exactly the kind of behavior you want from a useful benchmark.

## 2. Markov-1 is the current “best shallow explanation”

Right now, the best simple explanation of the Noita trigram-token stream is:

* shallow observed-token sequential dependence

That does **not** mean the puzzle is Markovian or solved.
It means that among the currently tested simple generative families, Markov-1 reproduces the benchmark best.

## 3. Higher-order observed-token Markov models are not a productive frontier

The diagnostics strongly suggest that the corpus is too small for sparse order-2 context modeling to be stable.

So the project should not spend much more effort on:

* Markov-3
* denser short-memory baselines
* increasingly fragile local n-gram generators

## 4. Stateful remapping is still worth pursuing

Even though `perm_state_v2` did not beat `markov1`, it did beat:

* `perm_state_v1`
* all Markov-2 variants
* i.i.d.

That is meaningful. It says:

* evolving remapping is not ridiculous
* but the current implementation is too simple to yet dominate shallow Markov structure

So the permutation-state direction remains viable.

---

## Important caveats about the current benchmark

## 1. Some benchmark components are still weak

Several shared-section metrics are still not very discriminative.

For example, some pairwise shared-structure terms appear to saturate or cluster too tightly across families.

That means the current total score should be treated as:

* useful for family ranking
* not yet a definitive scientific measure

## 2. The shallow baseline may benefit from being closer to the benchmark representation

`markov1` models the observed trigram-token stream directly.
That naturally gives it an advantage on local observed-token statistics.

By contrast, permutation-state families introduce an intermediate latent layer, which may only pay off if the state update and permutation structure are sufficiently expressive.

## 3. Matching the benchmark does not imply decryption

A family matching the corpus well only means:

* it reproduces the corpus’s statistical fingerprints better

It does **not** mean:

* the plaintext has been recovered
* the true cipher has been identified
* the model is historically or diegetically correct

This repo is a **family discrimination** framework first.

---

## Most important takeaways so far

### Strong conclusions

* The Noita trigram-token corpus is not well explained by a memoryless model.
* The corpus contains meaningful short-range structure.
* Sparse higher-order observed-token Markov modeling is not effective on this dataset.
* A benchmark-driven model-family comparison framework is now operational.

### Tentative conclusions

* Stateful evolving-remapping families are plausible but currently underpowered.
* The current permutation-state implementations are too simple to clearly outperform Markov-1.
* Better structured state updates or richer permutation mechanisms may still matter.

---

## Recommended next steps

## 1. Improve the benchmark before adding too many more families

This is probably the single most important next step.

Focus on making the harder components more informative, especially:

* pairwise shared exact n-gram metrics
* pairwise shared isomorph metrics
* maybe normalization schemes for cross-message overlap
* maybe local alignment style features

Right now the ranking is useful, but some structural features are not pulling their weight.

## 2. Add stronger permutation-state families

The best next model family is still stateful, but it should be more structured than v1/v2.

Promising directions:

* token-driven state update with richer but still low-variance rules
* structured permutation banks instead of simple rolled permutations
* block-rotation or swap-schedule permutations
* emission-driven or mixed update rules
* message-specific initial states with shared global mechanism

The key is to keep the family interpretable and low-parameter.

## 3. Add one or two sanity-check classical cipher families

Not because they are likely to win, but because they help calibrate the benchmark.

Good options:

* static substitution family
* Vigenère-like family
* simple autokey-like family

These would help answer:

* does the benchmark actually reject obvious static families?
* where do classical polyalphabetic families land relative to Markov and permutation-state models?

## 4. Add a component-wise analysis workflow

Instead of looking mostly at `total_weighted`, start building a routine that asks:

* which features does a family improve?
* which features does it worsen?
* which benchmark components actually distinguish families?

This will make model development much more principled.

## 5. Keep Markov-1 as the main baseline to beat

From now on, every new family should be evaluated against the question:

> Does this beat Markov-1, either overall or on the structural components we care about most?

That is now the practical benchmark bar.
