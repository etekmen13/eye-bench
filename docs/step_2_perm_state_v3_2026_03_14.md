# Step 2: Stronger Permutation-State Family

## Goal

The report recommended a stronger stateful family rather than more shallow observed-token Markov order. I implemented `perm_state_v3` as the next structured step beyond `perm_state_v2`.

## Design

`perm_state_v3` keeps the same broad modeling idea:

* latent source fitted as Markov-1
* observed tokens emitted through a hidden-state-dependent permutation

But I strengthened the family in three specific ways:

* `block_rotation` permutation bank instead of only globally rolled permutations
* mixed state update using both latent-token and emitted-token delta maps
* message-specific initial states generated by a shared cyclic schedule

The point was to increase structure without turning the family into an unconstrained random remapper.

## Devlog

I explicitly avoided a fully random high-variance v3. That would have increased nominal expressivity, but it would also have made failures harder to interpret. The chosen v3 stays low-parameter:

* one structured permutation bank
* two low-variance token-to-delta maps
* one message-initial-state schedule

This keeps the family interpretable enough that future tuning can still be principled.

I also kept the latent source as fitted Markov-1. That isolates whether the extra state mechanism helps, instead of mixing a stronger state model with a different latent source and making attribution unclear.

## Result

In the regenerated 50-seed ranking, `perm_state_v3` landed at:

* `3.128914` mean weighted distance

That puts it in the same general stateful band, but it did **not** beat:

* `markov1`
* `static_sub`
* `perm_state_v2`
* `perm_state_v1`
* `vigenere_like`

So the step is useful as family expansion, but not as a win. The practical conclusion is that richer remapping alone is still not enough; the state update rule likely matters more than the current v3 construction.
